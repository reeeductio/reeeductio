# Python SDK Architecture

This document explains the two-layer architecture of the reeeductio Python SDK.

## Overview

The SDK is split into two distinct layers:

```
python-sdk/
├── reeeductio_client/     # Layer 1: Auto-generated low-level client
│   ├── api/               # API endpoint functions
│   ├── models/            # Pydantic models
│   └── client.py          # HTTP clients
│
└── reeeductio/            # Layer 2: Hand-written high-level SDK
    ├── __init__.py        # Public API exports
    ├── client.py          # Space client
    ├── auth.py            # Authentication helpers
    ├── crypto.py          # Cryptographic utilities
    └── messages.py        # Message handling
```

## Layer 1: Auto-Generated Client (`reeeductio_client`)

### Purpose
Provides a direct, type-safe interface to the reeeductio REST API.

### Generated By
`openapi-python-client` tool from [openapi.yaml](../openapi.yaml)

### Characteristics
- **Type-safe**: All request/response types are Pydantic models
- **Complete**: 1:1 mapping to every API endpoint
- **Sync & Async**: Four variants per endpoint (sync, sync_detailed, asyncio, asyncio_detailed)
- **Low-level**: No business logic, just HTTP calls and serialization

### Regeneration
When the OpenAPI spec changes:
```bash
openapi-python-client generate \
  --path ../openapi.yaml \
  --output-path . \
  --config ../openapi-python-client-config.yaml
```

### Usage Example
```python
from reeeductio_client import AuthenticatedClient
from reeeductio_client.api.state import get_spaces_space_id_state_path

client = AuthenticatedClient(
    base_url="https://api.example.com",
    token="jwt-token",
)

state_entry = get_spaces_space_id_state_path.sync(
    client=client,
    space_id="...",
    path="profiles/alice",
)
```

### Key Files
- `client.py`: `Client` and `AuthenticatedClient` classes
- `api/`: One module per OpenAPI tag (authentication, state, messages, blobs, web_socket)
- `models/`: Pydantic models for all request/response bodies
- `types.py`: Common types like `Response`, `UNSET`

## Layer 2: High-Level SDK (`reeeductio`)

### Purpose
Provides convenient, developer-friendly abstractions over the low-level client.

### Hand-Written
This layer is manually maintained and version-controlled.

### Characteristics
- **Convenient**: Hides complexity (auth flows, signing, encoding)
- **Domain-focused**: Methods match user intent (e.g., `send_message`, `get_profile`)
- **Opinionated**: Makes sensible default choices
- **Independent**: Can evolve without regenerating

### Modules

#### `__init__.py`
Public API surface. Exports commonly used classes and functions:
```python
from reeeductio import Space, generate_keypair, Ed25519KeyPair
```

#### `client.py` - `Space` class
Main entry point for interacting with a space. Provides:
- Automatic authentication management
- State get/set/delete methods
- Message retrieval methods
- Blob upload/download methods
- Convenience methods (e.g., `get_profile`, `set_profile`)

#### `auth.py` - `AuthSession` class
Handles the challenge-response authentication flow:
- Request challenge from server
- Sign challenge with Ed25519 private key
- Verify signature and obtain JWT
- Automatic token refresh

#### `crypto.py`
Cryptographic utilities using the `cryptography` library:
- Ed25519 keypair generation
- Signing and signature verification
- SHA256 hashing
- Base64 encoding/decoding (standard and URL-safe)
- Typed identifier helpers (with header bytes)

#### `messages.py`
Message handling helpers:
- Message hash computation (for chain validation)
- Message posting with automatic signing
- Chain validation
- Placeholder for encryption/decryption (to be implemented)

### Usage Example
```python
from reeeductio import Space, generate_keypair

# High-level: simple and convenient
keypair = generate_keypair()
space = Space(space_id="...", keypair=keypair)

# Authentication happens automatically
space.set_profile(user_id, {"name": "Alice"})
messages = space.get_messages("general")
```

## Design Principles

### 1. Separation of Concerns
- **Low-level**: Generated code stays pure, never manually edited
- **High-level**: Business logic, convenience, and workflows

### 2. Progressive Disclosure
Users can:
- Start with high-level SDK for common tasks
- Drop to low-level client for advanced use cases
- Mix both layers as needed

### 3. Maintainability
- API changes only require regenerating Layer 1
- Layer 2 updates are surgical and version-controlled
- Clear boundaries prevent merge conflicts

### 4. Type Safety Throughout
- Pydantic models in Layer 1 provide runtime validation
- Type hints in Layer 2 provide IDE support
- Full `mypy` compatibility (when type stubs are added)

## When to Use Each Layer

### Use High-Level SDK (`reeeductio`) When:
- Building applications
- You want convenience and sensible defaults
- You're new to the API
- You want authentication handled automatically
- You need common workflows (profiles, messages, etc.)

### Use Low-Level Client (`reeeductio_client`) When:
- You need precise control over requests
- You're implementing advanced features
- You want to customize retry/timeout behavior
- You need access to raw response data
- You're building your own higher-level abstractions

## Future Enhancements

### Layer 1 (Generated)
- WebSocket client support (currently manual)
- Better type stubs for IDE autocomplete
- Configurable retry policies

### Layer 2 (Hand-Written)
- **Message encryption/decryption**: Implement using `cryptography` library (AES-GCM)
- **WebSocket integration**: Real-time message streaming
- **Capability helpers**: Grant/revoke capabilities, role management
- **Batch operations**: Efficient bulk state updates
- **Caching layer**: Optional client-side caching for state/messages
- **Async variants**: Async versions of Space class methods

## Dependencies

Both layers share core dependencies:
- `httpx`: HTTP client (sync and async)
- `attrs`: Dataclass-like structures
- `python-dateutil`: Date/time parsing
- `cryptography`: Ed25519, hashing (Layer 2 only)

## Testing Strategy

### Layer 1 (Generated)
- Tested implicitly through Layer 2 tests
- Validated against OpenAPI spec during generation

### Layer 2 (Hand-Written)
- Unit tests for crypto utilities
- Integration tests against running server
- Example scripts as smoke tests

## Versioning

- **Layer 1**: Version tracks OpenAPI spec version
- **Layer 2**: Semantic versioning independent of Layer 1
- Both distributed together in single package

## Contributing

### When Updating OpenAPI Spec
1. Update `openapi.yaml`
2. Regenerate Layer 1: `openapi-python-client generate ...`
3. Review changes in `reeeductio_client/`
4. Update Layer 2 if new endpoints need high-level wrappers
5. Update examples and documentation

### When Adding High-Level Features
1. Identify common use case or workflow
2. Implement in appropriate `reeeductio/` module
3. Export from `__init__.py` if public API
4. Add example usage
5. Update README
